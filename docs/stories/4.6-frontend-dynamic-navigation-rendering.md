**As a** 官网访客,
**I want** 导航栏根据后台栏目配置 API 动态生成，包括一级栏目和二级下拉菜单,
**so that** 管理员在后台调整栏目后，官网导航能自动更新，无需修改前端代码或重新部署。

## Epic Context

**Epic**: 4 - 内容管理基础

完成栏目管理、富文本编辑器集成、轮播图管理、内容发布三模式框架及前端动态导航

**Story Deliverables**:
- 前端导航 TypeScript 类型定义
- `useNavigation` composable（API 获取 + 缓存策略）
- 改造 AppHeader.vue 为 API 驱动（移除硬编码 navItems）
- API 响应缓存机制保证加载性能

### AC1: 导航栏数据从栏目配置 API 获取

**Scenario**
```gherkin
GIVEN 前端官网页面加载
WHEN AppHeader 组件渲染
THEN
  - 导航栏数据从公开栏目 API GET /content/v1/portal/columns 获取
  - 导航栏按 API 返回的 sortOrder 顺序显示一级栏目
  - 每个栏目显示 name 字段内容，链接到 routePath 指定的路由
  - "管理员登录"链接保留为独立的外部链接（不从 API 获取）
  - API 请求失败时降级为兜底静态导航数据
```

**Business Rules**

| ID | Rule |
|----|------|
| BR-1.1 | 导航数据通过 `useNavigation` composable 统一管理，不在组件内直接调用 API |
| BR-1.2 | SSR 模式下服务端首次渲染时获取数据，客户端 hydration 复用 |
| BR-1.3 | API 字段映射：name → 显示文本, routePath → 链接路径, subColumns → 二级菜单 |
| BR-1.4 | "管理员登录"不属于栏目数据，在组件中单独处理（保留现有逻辑） |
| BR-1.5 | API 失败时使用 FALLBACK_NAV_ITEMS 兜底数据（与当前硬编码数据一致） |

**Error Handling**

| Scenario | Code | Message | Action |
|----------|------|---------|--------|
| API 请求超时 | - | 网络超时 | 使用兜底静态数据 |
| API 返回空数组 | 200 | 无栏目数据 | 使用兜底静态数据 |
| API 返回 500 | 500 | 服务端错误 | 使用兜底静态数据 |

### AC2: 隐藏状态的栏目不在导航栏显示

**Scenario**
```gherkin
GIVEN 管理员在后台将某栏目设为隐藏（is_visible=0）
WHEN 官网前端获取栏目数据
THEN
  - 隐藏的栏目不出现在导航栏中
  - 导航栏仅显示 API 返回的可见栏目（API 已过滤 is_visible=0）
  - 不需要前端额外过滤，依赖后端 API 的过滤逻辑
```

**Business Rules**

| ID | Rule |
|----|------|
| BR-2.1 | 公开 API 已保证仅返回 is_visible=1 的栏目，前端无需二次过滤 |
| BR-2.2 | 如果所有栏目都被隐藏（API 返回空数组），使用兜底数据 |

---

### AC3: 二级菜单按配置的排序展示

**Scenario**
```gherkin
GIVEN 某一级栏目启用了二级菜单（subMenuEnabled=true）且有 subColumns 数据
WHEN 用户 hover 该栏目
THEN
  - 下拉菜单展示 subColumns 中的所有二级菜单项
  - 二级菜单项按 sortOrder 升序排列（API 已排序）
  - 二级菜单项显示 name 字段，链接路由为 routePath + ?category={categoryCode}
  - 未启用二级菜单的栏目（subMenuEnabled=false 或 subColumns 为空）不显示下拉箭头和下拉菜单
```

**Business Rules**

| ID | Rule |
|----|------|
| BR-3.1 | 二级菜单路由格式：`{父栏目 routePath}?category={subColumn.categoryCode}` |
| BR-3.2 | subColumns 为空数组或 subMenuEnabled=false 时，该栏目按普通链接渲染（无箭头无下拉） |
| BR-3.3 | API 已保证 subColumns 按 sortOrder 升序，前端不需要额外排序 |
| BR-3.4 | 二级菜单的交互行为（hover 展开/收起/动画）保持 Story 3.1 已实现的样式不变 |

**UI Interaction**

| Trigger | Behavior |
|---------|----------|
| hover 有二级菜单的栏目 | 展开下拉菜单 + 箭头旋转（复用现有 CSS） |
| 鼠标移出 | 收起下拉菜单 + 箭头恢复 |
| 点击二级菜单项 | 跳转到 `{routePath}?category={categoryCode}` |
| 点击一级栏目名称 | 跳转到 `{routePath}`（无 query 参数） |

### AC4: 栏目配置变更后前端导航自动更新

**Scenario**
```gherkin
GIVEN 管理员在后台修改了栏目配置（名称/排序/可见性/二级菜单）
WHEN 用户刷新官网页面或首次访问
THEN
  - 导航栏显示最新的栏目配置
  - 无需修改前端代码或重新部署
  - SSR 模式下每次服务端渲染均获取最新数据
```

**Business Rules**

| ID | Rule |
|----|------|
| BR-4.1 | SSR 每次请求时服务端调用 API 获取最新数据，保证首屏始终最新 |
| BR-4.2 | 客户端导航（SPA 路由切换）时使用缓存数据，不重复请求 |
| BR-4.3 | 不依赖构建时数据或静态配置文件，完全由运行时 API 驱动 |

---

### AC5: API 响应缓存以保证加载性能

**Scenario**
```gherkin
GIVEN 官网导航栏在每个页面都需要渲染
WHEN 用户在站内多次导航切换页面
THEN
  - 导航数据在 SSR hydration 后客户端缓存复用，不重复请求 API
  - 客户端 SPA 路由切换时导航数据来自缓存，无额外网络请求
  - 缓存数据通过 Nuxt useFetch 的 key 机制去重
```

**Business Rules**

| ID | Rule |
|----|------|
| BR-5.1 | 使用 Nuxt `useFetch` + 固定 key `'nav-columns'` 实现请求去重和 SSR/客户端缓存 |
| BR-5.2 | SSR hydration 后客户端自动复用服务端数据，不触发额外请求 |
| BR-5.3 | 通过 `getCachedData` 选项实现客户端缓存复用，避免 SPA 路由切换时重复请求 |
| BR-5.4 | 缓存策略：SSR 每次渲染获取新数据；客户端在同一会话内缓存（刷新页面时由 SSR 更新） |

### Data Models

**公开栏目 API 响应结构** (`GET /content/v1/portal/columns`):

```typescript
// API 返回格式
interface ApiResult<T> {
  code: number
  message: string
  data: T
}

// 一级栏目
interface ColumnNav {
  id: number
  name: string        // 栏目显示名称
  code: string        // 栏目代码（home, intro, policy, schools, resources...）
  routePath: string   // 路由路径（/, /intro, /policy, /schools, /resources...）
  sortOrder: number   // 排序号
  subMenuEnabled: boolean  // 是否启用二级菜单
  subColumns: SubColumnNav[]  // 二级菜单列表（subMenuEnabled=false 时为空数组）
}

// 二级栏目
interface SubColumnNav {
  id: number
  name: string         // 二级菜单名称（国家政策、省级政策...）
  code: string         // 代码（national, provincial...）
  categoryCode: string // 分类代码（用于 query 参数）
  sortOrder: number    // 排序号
}
```

**NavItem 映射逻辑**:

```typescript
// ColumnNav → NavItem 映射
interface NavItem {
  label: string    // ← column.name
  path: string     // ← column.routePath
  children?: {
    label: string  // ← subColumn.name
    path: string   // ← `${column.routePath}?category=${subColumn.categoryCode}`
  }[]
}
```
