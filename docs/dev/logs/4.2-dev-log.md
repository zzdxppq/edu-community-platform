## Key Design Decisions

1. **`@Lazy` injection**: Used `@Lazy` on `SubColumnService` in `ColumnServiceImpl` to break circular dependency (Column needs SubColumn for hasChildren, SubColumn needs Column for parent validation). This avoids creating a third service or restructuring.

2. **Static converter pattern**: `SubColumnConverter` uses static utility methods matching `ColumnConverter` pattern, as recommended by architect review (no MapStruct in content-service).

3. **Nested resource REST pattern**: `/v1/columns/{columnId}/sub-columns` follows RESTful nested resource convention. The columnId is validated in service layer (parent must exist and own the sub-column).

4. **System preset protection granularity**: System presets (is_system=1) allow name and sortOrder modification but block code, categoryCode, and deletion. This balances customization needs with data integrity.

5. **Schools column hardcoded restriction**: `code='schools'` check prevents sub-column creation for the schools column, matching business requirement that this column has special rendering. Implemented as a simple string check in service layer.

6. **Soft delete name/code reuse**: Uniqueness checks use `@TableLogic` filtered queries, allowing soft-deleted records' names/codes to be reused for new records.
