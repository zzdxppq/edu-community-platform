## 技术决策

### 1. Token 刷新使用独立 axios 实例
refreshClient = axios.create() 无拦截器。避免 refresh 请求经过 service 拦截器导致死循环（401 → refresh → 401 → refresh ...）。

### 2. localStorage key 命名规范
access_token / refresh_token / user_info（下划线命名），与旧 'token' key 不兼容（故意 breaking change — Story 2.3 是首次实际登录实现，无历史数据迁移需要）。

### 3. 并发 401 处理
isRefreshing 标志 + failedQueue 数组。第一个 401 触发 refresh 请求，后续 401 排队等待。refresh 成功后 processQueue 释放所有排队请求，失败则全部 reject + logout。

### 4. Binding 2 设计偏差
Story 定义 request.ts 应 import auth.ts 的 refreshTokenApi，但实际使用 refreshClient.post（独立实例）。原因：auth.ts 的 refreshTokenApi 使用 service（含拦截器），会触发循环。设计偏差为有意为之。

### 5. 身份选择为纯 UI
el-radio-group selectedRole 不发送到后端 — 后端根据 admin_users.role 确定实际角色，前端选择仅为 UX 引导。
